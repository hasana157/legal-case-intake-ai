1. EXECUTIVE SUMMARY & PROJECT OVERVIEW
The Opposing-Argument Simulator is a generative AI application designed to provide
self-represented litigants with realistic adversarial case preparation through jurisdiction-grounded
retrieval-augmented generation (RAG). By integrating domain-specific legal knowledge bases with
advanced language models, this system simulates the counterarguments, procedural objections, and
counterevidence an opposing party would likely present—enabling laypeople to rehearse their cases
before hearing and dramatically improving preparedness and equity in the civil justice system.
Core Innovation:
Traditional legal-tech tools focus on one-directional support (document assembly, procedure
explanation, case law search). This system is the first to combine RAG-grounded retrieval with
adversarial role-play prompting, generating the strongest possible counterarguments a self-represented
litigant must anticipate. Every citation is verified against retrieved legal sources, ensuring
hallucination-free output suitable for real court preparation.
Key Deliverables:
 Jurisdiction-grounded RAG pipeline using Harvard Caselaw Access Project (CAP) dataset
 Adversarial LLM engine generating defensible counter-arguments with strict citation verification
 Serverless-optimized web application with SSE streaming to bypass timeout constraints
 Interactive case rehearsal interface with rebuttal preparation workspace
 Evaluation framework measuring citation accuracy, response latency, and user preparedness

2. PROBLEM STATEMENT & OBJECTIVES
The Justice Access Gap
Millions of self-represented litigants appear in U.S. civil courts annually—primarily in small claims,
tenancy, and family law matters. These individuals must simultaneously absorb complex procedural
rules, substantive legal standards, and evidence considerations, while entering hearings blind to the
specific counterarguments and precedents opposing counsel intends to raise. This structural
asymmetry of preparation—not legal merit—is a well-documented driver of adverse outcomes.
Why Existing Solutions Fall Short

counterevidence
 General-purpose LLMs hallucinate statutes and cite non-existent case law when used without
jurisdictional grounding
 Laypeople cannot assess the strength of their position relative to the adversary's expected response
 Legal tech is heavily skewed toward document assembly rather than case preparation and rehearsal

O2: Jurisdiction-Grounded Retrieval — Build a RAG pipeline ensuring every argument is traceable
to verified statutory or case-law sources from the target jurisdiction
O3: Adversarial Simulation — Generate realistic counter-arguments, procedural objections, and
counterevidence grounded strictly in retrieved legal authority
O4: Rebuttal Preparation & Safety — Enable users to draft counter-rebuttals while maintaining

4. RETRIEVAL-AUGMENTED GENERATION (RAG) ARCHITECTURE
4.1 RAG System Overview
The RAG architecture is the cornerstone of hallucination-free legal simulation. Rather than relying on
the language model's pre-trained knowledge (which may be outdated, unverifiable, or jurisdictionally
irrelevant), RAG retrieves relevant legal authorities before generation. This transforms the task from
'generate legal arguments' to 'synthesize retrieved legal authorities into persuasive arguments.'
4.2 RAG Pipeline Stages
Stage 1: Document Ingestion & Normalization
The Harvard Caselaw Access Project (CAP) dataset is streamed from Hugging Face and segmented
by court opinion. Each opinion is parsed to extract case metadata (case name, date, jurisdiction, court
level, headnotes) and opinion text. Documents are deduplicated and validated.
Stage 2: Chunking & Contextualization
Opinions are chunked by logical sections (e.g., Holding, Reasoning, Dissent) rather than arbitrary
tokens. Each chunk retains citation metadata and source reference. Chunk size: 300-500 tokens, with
50-token overlap to preserve context.
Stage 3: Embedding Generation
All chunks are encoded using a lightweight, domain-optimized embedding model (e.g.,
sentence-transformers all-MiniLM-L6-v2 or legal-BERT). Embeddings are 384-dimensional vectors
suitable for efficient similarity search. Embedding generation is parallelized to handle 100k+
documents.
Stage 4: Vector Database Storage
Embeddings are stored in Qdrant or Chroma with metadata indices for jurisdiction, case type, decision
date, and source citation. Hybrid search combines semantic similarity with metadata filters (e.g., 'only
retrieve cases from [target jurisdiction] after [relevant date]').
Stage 5: Retrieval & Re-Ranking
User case facts trigger a multi-pass retrieval: (1) semantic search for procedural and substantive
relevance, (2) BM25 keyword search for statute/rule references, (3) cross-encoder re-ranking to identify
top-5 most relevant authorities. Retrieved chunks are ranked by relevance score and source authority
(e.g., Supreme Court > Circuit Court > District Court).
Stage 6: Prompt Context Assembly

Retrieved authorities are formatted into the LLM prompt as 'LEGAL CONTEXT' blocks, each tagged
with source citation. The prompt explicitly instructs the model to reference only these retrieved sources
and to cite every claim.

5. GENERATIVE AI & ADVERSARIAL PROMPTING STRATEGY
5.1 LLM Model Selection
The system uses Claude 3.5 Sonnet (or equivalent state-of-the-art model) for adversarial argument
synthesis. This model is selected for: (1) superior reasoning and argument construction, (2) strong
instruction-following for role-play constraints, (3) native support for streaming via SSE for timeout
mitigation, (4) documented safety and hallucination resistance with grounded context.
5.2 Adversarial Prompting Framework
The system prompt is engineered to enforce strict adversarial role-play while constraining outputs to
retrieved sources:
SYSTEM PROMPT TEMPLATE (Pseudocode):
"You are an experienced opposing counsel preparing a defense against a 
[CLAIM_TYPE] claim in [JURISDICTION] court. Your goal is to construct 
the strongest possible counter-arguments, procedural objections, and 
counterevidence using ONLY the legal authorities below.
LEGAL CONTEXT:
[Retrieved case law and statutes with citations]
LITIGANT'S CASE SUMMARY:
[Structured case facts]
Your response must:
1. Highlight 3-5 critical vulnerabilities in the plaintiff's case
2. Cite specific cases, statutes, and procedural rules supporting each defense
3. Identify procedural defects (e.g., notice failures, jurisdictional issues)
4. Suggest likely discovery disputes or evidentiary objections
5. Assign confidence scores (High/Medium/Low) to each argument
6. Include footnote citations to retrieved sources
Do NOT invent cases, statutes, or authorities. If you cannot support 
a point with the retrieved context, explicitly state so."
5.3 Output Formatting & Verification
 LLM outputs structured JSON with keys: counter_arguments[], procedural_defenses[],
counterevidence[], citations[]
 Each counter-argument includes: claim_text, supporting_authority[], confidence_score
 Citations are parsed via regex to extract case names and statute references
 Citation Verification Engine cross-references extracted citations against retrieved document chunks

 Unverified citations are flagged and removed; response regenerated if verification fails
catastrophically
5.4 Hallucination Prevention & Grounding Assurance
The Grounding Verification metric (Gv) rigorously tracks hallucination risk:
Grounding Verification Formula: Gv = Cv / Ct = 1.00
Where Cv = count of verified citations (matched to retrieved chunks via regex) and Ct = total citation
count in output. Target: Gv = 1.00 (100% verification). Any response with Gv < 0.95 is flagged for
manual review or regenerated.

6. VECTOR DATABASE & EMBEDDING STRATEGY
6.1 Embedding Model Selection
Primary (Recommended): sentence-transformers/all-MiniLM-L6-v2: 384-dimensional, fast (~100
docs/sec on CPU), strong legal domain performance via fine-tuning on legal corpora. Lightweight and
suitable for Colab/serverless.
Alternative (Enhanced): Legal-BERT or domain-optimized models: 768-dimensional, trained on legal
texts, superior jurisdiction-specific retrieval. Slower (~20 docs/sec) but higher precision for edge cases.
Fallback: OpenAI Text Embedding 3 (small): 1536-dimensional, API-based, production-grade quality.
Higher latency and cost; use for production if Colab-based pipeline proves insufficient.
6.2 Vector Database: Qdrant vs. Chroma
Qdrant (Recommended for Production):  Distributed, scalable, supports hybrid search (dense +
sparse vectors)  REST API for cloud deployment  Supports dynamic indexing and real-time updates 
Ideal for 100k+ document collections  Free cloud tier sufficient for MVP
Chroma (Recommended for Development/Colab):  Lightweight, in-process or client-server  Easy
integration with Langchain/LlamaIndex  Built-in metadata filtering  Sufficient for <50k documents 
Perfect for initial prototyping and Colab notebooks

8. DEPLOYMENT & PRODUCTION ARCHITECTURE
8.1 Vercel Serverless Functions & Timeout Mitigation
Standard Vercel hobby-tier serverless functions impose a 15-second execution timeout. RAG
operations—particularly vector similarity search over 100k+ documents followed by LLM token
generation—frequently exceed this threshold. The solution is Serverless Event Streaming (SSE):
Timeout Mitigation Strategy (SSE Pattern):
1. User submits case facts to /api/generate-opposition
2. Backend immediately returns a streaming response (HTTP 200 with Transfer-Encoding: chunked)
3. Vector search and LLM generation proceed on the open stream
4. Tokens are written to the stream instantly as generated (e.g., every 100ms)
5. Each token write resets the connection timeout clock
6. Frontend continuously reads stream buffer, rendering text blocks as they arrive
7. Total response latency: ~3-5 seconds, well below 15s timeout
This pattern allows LLM completions of 2000+ tokens within serverless constraints.

9. EVALUATION METRICS & QUALITY ASSURANCE
9.1 Citation Accuracy Evaluation (Hallucination Detection)
Grounding Verification (Gv)
Primary metric measuring hallucination risk. For each generated response:
 Extract all case names and statute citations from LLM output via regex
 Cross-reference against retrieved document metadata and chunk text
 Count verified citations (Cv) vs. total citations (Ct)
 Gv = Cv/Ct — target ≥ 0.95 (95% verification minimum)
 Log unverified citations for manual review (potential hallucinations)
RAGAS Framework Integration
Implement RAGAS (Retrieval-Augmented Generation Assessment) metrics to evaluate retrieval and
generation quality independently:
 Faithfulness: Does generated argument align with retrieved legal sources? (Target: >0.80)
 Answer Relevance: Does output directly address case facts? (Target: >0.85)
 Context Relevance: Are retrieved documents actually pertinent? (Target: >0.75)
 Context Precision: What fraction of retrieved contexts are relevant? (Target: >0.70)
9.2 Retrieval Performance Metrics
Mean Reciprocal Rank (MRR): Position of first relevant document in retrieval ranking. Target: MRR
>0.70
Normalized Discounted Cumulative Gain (NDCG): Ranking quality accounting for relevance grades.
Target: NDCG@5 >0.65
Latency: Time from case query to top-k retrieval. Target: <500ms for 100k document corpus
Recall: Proportion of all relevant documents retrieved. Target: Recall@20 >0.60

9.3 Generation Quality & Latency
BLEU / ROUGE Scores: Measure textual overlap with gold-standard opposing arguments drafted by
legal experts. (Reference-based evaluation)
Token Latency: Time to first token (TTFT): <300ms. Average token latency: <50ms. Total streaming
latency: <3.0s for 500-token response.
Output Length: Target: 400-800 tokens per opposing argument (sufficient for comprehensive but
focused rebuttal preparation).
Argument Diversity: Number of distinct counter-argument categories identified. Target: ≥3 per case.
9.4 User-Level Evaluation (A/B Testing)
 Recruit self-represented litigants preparing for real hearings
 Randomize: Control group (no simulator) vs. Treatment group (with simulator)
 Measure outcomes: Hearing preparedness self-assessment, argument anticipation accuracy,
perceived confidence
 Secondary outcome: Post-hearing survey on argument quality and decision impact
 Target: Treatment group shows statistically significant improvement in preparedness (p < 0.05)

10. SAFETY FRAMEWORKS & LEGAL SAFEGUARDS
10.1 Hallucination Prevention & Verification Layers
Layer 1: Prompt Constraint: System prompt explicitly forbids citation of non-retrieved authorities. LLM
is instructed: 'If you cannot find supporting authority in the LEGAL CONTEXT, explicitly state so.'
Layer 2: Citation Extraction & Verification: All citations extracted via deterministic regex matching.
Each citation string cross-referenced against retrieved document metadata and text content.
Layer 3: Grounding Validation (Gv): Every response scored on Grounding Verification metric.
Responses with Gv <0.95 are flagged for regeneration or manual review.
Layer 4: Output Filtering: Automatic removal of any unverified citations before user presentation.
Fallback: regenerate response with lower k (fewer retrieved contexts) to simplify reasoning.
Layer 5: Human Review Audit Trail: All low-confidence responses (<0.80 Grounding Verification)
logged for periodic human legal expert review. Findings inform prompt refinement.

11.1 Data Ingestion Pipeline (Google Colab)
# Harvard CAP Dataset Streaming in Colab
from datasets import load_dataset
from langchain.text_splitter import RecursiveCharacterTextSplitter
from sentence_transformers import SentenceTransformer
import qdrant_client
# 1. Stream CAP dataset from Hugging Face (avoids OOM)
dataset = load_dataset(
    "free-law/Caselaw_Access_Project", 
    split="train", 
    streaming=True  # Critical: prevents loading entire dataset into RAM
)
# 2. Initialize embedding model (lightweight, fast)
model = SentenceTransformer('all-MiniLM-L6-v2')  # 384-dim, ~10MB
# 3. Streaming ingestion loop
client = qdrant_client.QdrantClient(":memory:")  # Local for testing
collection_name = "legal_opinions"
text_splitter = RecursiveCharacterTextSplitter(
    chunk_size=400,
    chunk_overlap=50,
    separators=["\n\n", "\n", " ", ""]
)
for idx, case_record in enumerate(dataset.take(50000)):  # Stream 50k cases
    # Extract metadata
    case_name = case_record.get('name', 'Unknown')
    citation = case_record.get('citation', '')
    decision_date = case_record.get('decision_date', '')
    opinion_text = case_record.get('opinion', '')
    
    # Split into chunks
    chunks = text_splitter.split_text(opinion_text)
    
    # Generate embeddings and store
    for chunk_idx, chunk in enumerate(chunks):
        embedding = model.encode(chunk)
        
        client.upsert(
            collection_name=collection_name,
            points=[{
                "id": idx * 1000 + chunk_idx,
                "vector": embedding.tolist(),
                "payload": {
                    "case_name": case_name,
                    "citation": citation,
                    "decision_date": decision_date,

                    "chunk_text": chunk,
                    "chunk_index": chunk_idx
                }
            }]
        )
    
    if (idx + 1) % 1000 == 0:
        print(f"Processed {idx + 1} cases...")
print("Ingestion complete!")
11.2 FastAPI Endpoint: RAG-Grounded Generation with SSE
# FastAPI route with Serverless Event Streaming (SSE)
from fastapi import FastAPI, HTTPException
from fastapi.responses import StreamingResponse
import asyncio
from anthropic import Anthropic
app = FastAPI()
client = Anthropic()
@app.post("/api/generate-opposition")
async def generate_opposition(case_data: dict):
    """
    Generate opposing arguments via RAG-grounded LLM
    Streams response to bypass Vercel 15s timeout
    """
    
    # 1. Extract case facts
    jurisdiction = case_data.get('jurisdiction')
    claim_type = case_data.get('claim_type')
    case_facts = case_data.get('case_facts')
    
    # 2. Retrieve relevant legal authorities via vector search
    query_embedding = embed_model.encode(case_facts)
    retrieved_cases = qdrant_client.search(
        collection_name="legal_opinions",
        query_vector=query_embedding.tolist(),
        limit=10,  # Top-10 most relevant
        score_threshold=0.60
    )
    
    # 3. Format legal context for prompt
    legal_context = "\n\n".join([
        f"Case: {case['payload']['case_name']}\n"
        f"Citation: {case['payload']['citation']}\n"
        f"Text: {case['payload']['chunk_text']}"
        for case in retrieved_cases
    ])
    
    # 4. Build system prompt
    system_prompt = f"""You are experienced opposing counsel in a {claim_type} 
case in {jurisdiction} jurisdiction. Generate the strongest counter-arguments 

using ONLY the legal authorities provided below.
LEGAL CONTEXT:
{legal_context}
Respond with 3-5 key counter-arguments, each with:
- Claim description
- Supporting case citation(s)
- Confidence level (High/Medium/Low)
Do NOT invent cases or statutes. Only cite retrieved authorities."""
    
    # 5. Stream response from Claude API
    async def stream_response():
        with client.messages.stream(
            model="claude-3-5-sonnet-20241022",
            max_tokens=1000,
            messages=[{"role": "user", "content": case_facts}],
            system=system_prompt
        ) as stream:
            for text in stream.text_stream:
                yield f"data: {json.dumps({'delta': {'text': text}})}\n\n"
                await asyncio.sleep(0)  # Yield control to event loop
    
    return StreamingResponse(
        stream_response(),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no"
        }
    )

                body: JSON.stringify(caseData)
            });
            if (!response.ok) throw new Error('API error');

            const reader = response.body.getReader();
            const decoder = new TextDecoder();
            while (true) {
                const { done, value } = await reader.read();
                if (done) break;
                const text = decoder.decode(value);
                const lines = text.split('\n');
                for (const line of lines) {
                    if (line.startsWith('data: ')) {
                        const data = JSON.parse(line.slice(6));
                        setOpposingArguments(prev => 
                            prev + (data.delta?.text || '')
                        );
                    }
                }
            }
        } finally {
            setIsStreaming(false);
        }
    };
    return (
        <div className="simulation-panel">
            <h2>Opposing Arguments (Simulated)</h2>
            <div className="streaming-text">
                {opposingArguments}
                {isStreaming && <span className="cursor">|</span>}
            </div>
            <div className="rebuttal-panel">
                <textarea 
                    placeholder="Enter your counter-argument..."
                    disabled={isStreaming}
                />
            </div>
        </div>
    );
}

12. APPENDICES
12.1 Glossary of Key Terms
RAG (Retrieval-Augmented Generation): Technique combining document retrieval with LLM
generation. Retrieved documents ground the model's response, reducing hallucinations.
Vector Database: Database storing high-dimensional embeddings and enabling semantic similarity
search.
Embedding: Dense numerical representation (e.g., 384-dimensional vector) of text capturing semantic
meaning.
Hallucination: LLM generating plausible but false information, e.g., fabricated case citations.
SSE (Serverless Event Streaming): HTTP streaming technique allowing continuous data transfer
without request timeout.
Grounding Verification (Gv): Metric measuring fraction of LLM citations verified against retrieved
sources.
Self-Represented Litigant (SRL): Individual appearing in court without attorney representation (pro
se).
Jurisdiction-Grounded: Retrieval and generation constrained to legal authorities from specific
geographic or judicial jurisdiction.
